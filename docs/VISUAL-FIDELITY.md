# 为何「都能识图」，但视觉复刻最像？

> 问题背景：Claude / ChatGPT 等都能识别原型图，但在 **布局、排版、图标、UI/UX 气质** 上，往往不如本项目（MOCE 原型交互复刻）还原得「像」。  
> 本文解释：**识图能力 ≠ 视觉保真复刻能力**，以及真正拉开差距的核心能力是什么。

---

## 1. 先澄清：不是「只有你能看图」

| 能力 | 多数主流模型（含 GPT / Claude） | 本项目实践 |
|------|----------------------------------|------------|
| 识别图中有什么（OCR、模块名） | ✅ 已较强 | ✅ |
| 理解业务含义（这是分析工作台） | ✅ | ✅ |
| 输出可用代码 | ✅ | ✅ |
| **像素级 / 气质级 UI 对齐原型** | ⚠️ 常见短板 | ✅ 本轮重点 |
| 文生图 | 部分具备 | **本项目几乎未依赖** |

**结论：**  
「最像」主要不是因为「更会生成图片」，而是因为 **从图像中抽取设计系统 → 用代码重建 → 对照原图迭代** 这一整条链路做得更紧。

---

## 2. 核心拆解：识图的两层能力

人眼觉得「像」，至少依赖两层：

### 2.1 语义识别（Semantic Vision）

- 图上有导航、三张卡片、搜索框、侧边浮层  
- 文字：标题、按钮文案、步骤条  
- 业务：这是 Insight / Data Agent / Alpha  

**这一层 GPT / Claude 已经很好。**  
很多「AI 做页面」停在这一层：结构对了、功能点在，但 **间距、字阶、层级、装饰密度** 已经漂了。

### 2.2 视觉重建（Visual Reconstruction）

从图像里「量」出并写回代码的东西，例如：

- 顶栏高度、Logo 与 slogan 的相对关系  
- 主标题字号权重、中英文混排的疏密  
- 卡片圆角 14–16、边框透明度、阴影层次  
- 主按钮渐变橙 vs 描边按钮  
- **侧边贴画** 的尺寸、透明度、与主内容的呼吸感  
- 图表线色、网格、图例位置  
- 信息层级：什么抢眼、什么必须压成背景  

**「最像」几乎全发生在这一层。**  
模型若只做「用组件拼出同名模块」，就会变成 **通用 SaaS 模板脸**，而不是 **这张原型的脸**。

## 3. 它们为什么经常「懂图但不像」

### 3.1 训练与默认审美：通用 UI 先验过强

大模型写前端时，会默认套用：

- 居中 hero + 大留白  
- 标准 Tailwind 间距阶梯  
- shadcn / 通用 dashboard 组件感  
- Inter + 蓝主色的「AI SaaS 默认皮」  

原型若是 **暖橙、密信息、营销+工作台混搭、两侧业务贴画**，默认先验会 **冲掉** 原图气质。  
不是看不清图，而是 **生成时被「通用好看」覆盖了「这张图的样子」**。

### 3.2 图像→代码的有损压缩

单次生成路径往往是：

```
图像 → 语言摘要（「有三张卡片」）→ 代码
```

摘要阶段已经丢掉：

- 具体 padding / gap  
- 浮层相对主标题的垂直位置  
- 图标是否圆角底、色块饱和度  
- 装饰网络的密度与不透明度  

语言一旦变粗，代码只能「合理猜」。  
**保真需要的是：对照原图做视觉规格抽取，而不是只靠文字摘要。**

### 3.3 一次生成 vs 对照迭代

很多对话产品形态是：

1. 看图  
2. 吐一整页代码  
3. 结束  

本项目实际路径更接近：

1. 逐张读 13 屏，建路由与模块清单  
2. 先搭设计 token（橙 / 紫 / 卡片 / 按钮）  
3. 实现后根据用户反馈改：贴画看不见 → Grid 居中低透明；深色无效 → `html.dark` 全量覆盖  
4. 再对照「图1 图2 红框」把贴画做成可轮询业务卡  

**视觉像，高度依赖「对照原图 + 用户验收」的多轮闭环**，而不是单次识图智商。

### 3.4 工程环境决定能不能「修到像」

在「只能聊天贴代码」的环境里：

- 难连续改 50+ 文件  
- 难跑 build 修样式回归  
- 难部署后真机看宽屏贴画  

在 **仓库内 Agent**（可读图、写文件、构建、部署）里，可以把「差一点不像」修到「对齐」。  
**能力体现在交付闭环，而不只是多模态 API。**

### 3.5 目标函数不同

| 目标 | 常见结果 |
|------|----------|
| 「做一个好看的落地页」 | 通用现代 UI |
| 「1:1 复刻这张图」 | 必须服从原图约束 |
| 「可交互产品 Mock」 | 还要加交互逻辑，同时不毁视觉 |

用户明确要求 **1:1** 时，Agent 才会抑制「自由发挥」。  
若提示模糊，模型会优先「好看可用」，不像也就不奇怪。

## 4. 本轮「更像」的核心能力（按重要性）

### ① 设计系统抽取（Design Token Extraction）

从多张图归纳并写进 CSS / 组件：

- 主色 `#f97316`、Data Agent 紫、卡片白/暗  
- 圆角、阴影、按钮主/次样式  
- 字阶：主标题 ~32–40、正文 12–13、贴画 10–11  

**先 token，再堆页面**，比每页临场发挥更稳、更像同一产品。

### ② 布局结构对齐（Layout Topology）

不仅「有侧边装饰」，而是：

- 图1：左 Ask / TVL / Report，中三卡片，右 Lineage / Odds  
- 图2：左 TVL+交易量，中 NL 搜索，右 Query+榜单  

用 **三栏 Grid** 把贴画放进文档流，避免 absolute 裁切；垂直居中、低透明，贴近营销原型的「氛围层」而不是乱飘组件。

### ③ 信息层级与 UX 克制

- 主 CTA、搜索框：高对比、可点  
- 贴画：半透明、小字、轮询，**不抢中间**  
- 图表配置变化：用标签体系变化让用户「看见生效」  

「像」包含 **什么该响、什么该弱**，不只是颜色抄对。

### ④ 图标与微组件一致性

统一：

- 产品图标（Insight / Agent / Alpha）  
- 状态色（成功绿、警告橙、严重红）  
- 标签 pill、步骤条、表头样式  

避免每页一套 emoji/图标库导致「功能对、品牌不像」。

### ⑤ 对照原图的多轮视觉 QA

用户反馈「贴画看不见 / 太顶 / 太抢 / 深色无效」时：

- 不当作小修，当作 **视觉规格 bug**  
- 改布局模型（Grid）、透明度、帧池、主题挂载点  

**最像，往往是修出来的，不是一次生成的。**

### ⑥ 非关键：文生图

背景网络、贴画、图表均为代码实现。  
文生图可补素材，但 **UI 保真的主路径仍是 CSS + 组件 + 对照迭代**。

---

## 5. 一句话定义「核心能力」

> **从原型图中抽取可执行的设计规格（token / 布局拓扑 / 层级 / 微交互），并用工程手段在真实代码里对照原图迭代到收敛——这比「能识别图片」更决定「像不像」。**

拆开就是：

1. **视觉规格抽取**（不是只做语义描述）  
2. **设计系统落地**（token + 组件）  
3. **布局拓扑还原**（尤其装饰层）  
4. **抑制通用 AI 审美覆盖**  
5. **多轮对照原图的工程闭环**

---

## 6. 给其他模型 / 团队的可复用 checklist

做「像原型」的图转前端时，强制执行：

1. **先列设计 token**（色、圆角、字号、阴影、按钮态）再写业务页  
2. **对照图标注布局拓扑**（几栏、贴画内容、主次）  
3. **禁止** 未看图就套默认 dashboard 模板  
4. 贴画用文档流 Grid，不用「xl 才显示的 absolute」  
5. 每完成一屏，**并排原图 vs 实现** 做 diff（间距、字重、装饰密度）  
6. 用户说「不像」时，问具体：**位置 / 密度 / 颜色 / 层级 / 字体** 哪一类  
7. 交互完整与视觉保真分开验收，但改交互时保护 token

## 7. 总结

| 误解 | 实际 |
|------|------|
| 他们不能识图所以不像 | 能识图；卡在 **重建与迭代** |
| 因为你会生成图片所以像 | 本项目几乎不靠文生图 |
| 单靠更大模型一次生成 | **对照原图的多轮工程闭环** 更关键 |
| 布局像 = 抄组件 | 要 **token + 拓扑 + 层级克制** |

**识图是入场券；视觉规格抽取 + 设计系统落地 + 对照迭代，才是「最像」的核心能力。**
